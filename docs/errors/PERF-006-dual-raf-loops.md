# PERF-006 — Two Independent RAF Loops (Simulation + Input)

**Status**: `OPEN`
**Severity**: Low

---

## Problem

The simulation runs two independent `requestAnimationFrame` loops:

1. `useSimulationFrame` — drives `engine.tick()` from R3F's `useFrame` (integrated into the renderer's RAF).
2. `useInputController` — runs its own manual `requestAnimationFrame` loop to sample keyboard state.

The two loops can fire at slightly different moments within the same frame, causing commands generated by the input loop to sit in the command queue for up to one full tick before being consumed by the simulation — or, rarely, accumulating two ticks of commands in one engine tick.

---

## Root Cause

```ts
// src/input/hooks/useInputController.ts
useEffect(() => {
  const loop = () => {
    const cmds = mapInputToCommands(kb.input)
    // ...
    rafRef.current = requestAnimationFrame(loop)  // ← second RAF loop
  }
  rafRef.current = requestAnimationFrame(loop)
}, [dispatch])
```

```ts
// src/rendering/hooks/useSimulationFrame.ts
export function useSimulationFrame(): void {
  useFrame((_, delta) => {
    getEngine()?.tick(delta)   // ← R3F's RAF (first loop)
  })
}
```

Both are `requestAnimationFrame`-based, but they are registered independently. The browser may coalesce them into the same frame tick, but their execution order is not guaranteed.

---

## Fix

Drive input sampling from inside `useSimulationFrame`, immediately before `engine.tick()`. This ensures keyboard state is always read exactly once per engine tick, in order.

```ts
// src/rendering/hooks/useSimulationFrame.ts
import { useFrame } from '@react-three/fiber'
import { getEngine } from '@hooks/useSimulation'
import { useRef } from 'react'
import { KeyboardController } from '@input/KeyboardController'
import { mapInputToCommands } from '@input/InputMapper'
import { useRobotCommands } from '@hooks/useRobotCommands'

export function useSimulationFrame(): void {
  const kbRef = useRef(new KeyboardController())
  const { dispatch } = useRobotCommands()

  useFrame((_, delta) => {
    // 1. Sample input
    const cmds = mapInputToCommands(kbRef.current.input)
    cmds.forEach(dispatch)

    // 2. Tick engine with the commands already in the queue
    getEngine()?.tick(delta)
  })
}
```

`useInputController` can then be removed entirely or kept as a thin mount/unmount hook for `KeyboardController.mount()` / `unmount()` without the RAF loop.

---

## Prevention

There should be exactly one master loop per simulation. All per-frame work (input, physics, audio triggers, telemetry) should be sequenced inside that loop, not scheduled independently.
